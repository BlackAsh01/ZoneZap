\begin{abstract}
This paper presents ZoneZap, a mobile-based safety system designed to assist cognitively impaired patients, including those with Alzheimer's disease, dementia, and other cognitive disorders. The system integrates real-time location tracking and artificial intelligence-based anomaly detection to provide proactive safety monitoring. ZoneZap supports two roles: \textbf{users} (wards), who are monitored and can trigger emergency alerts and use reminders, and \textbf{guardians}, who receive panic and wandering alerts and can create reminders for their wards. The mobile application is built as a native Android app using Kotlin and Material Design; it uses the Fused Location Provider for GPS tracking and stores location data in Firebase Firestore. Firebase Cloud Functions process emergency alerts, analyze location patterns for wandering behavior (using speed and movement variance thresholds), and deliver overdue reminder notifications every five minutes via Firebase Cloud Messaging. An optional Python-based AI engine uses the Isolation Forest algorithm to train on movement data from Firestore and can be used for anomaly prediction. Firestore security rules enforce access control so that only the owning user or linked guardians can access alerts, reminders, and movement logs. The system addresses critical safety concerns for vulnerable populations while maintaining a scalable cloud architecture and clear separation between ward and guardian flows.
\end{abstract}

\section{Introduction}

Cognitive impairments, including Alzheimer's disease, dementia, and traumatic brain injuries, affect millions of individuals worldwide. One of the most significant safety concerns for this population is wandering behavior, which can lead to disorientation, injury, or even fatality. Traditional monitoring methods, such as constant supervision or physical restraints, are often impractical, expensive, or compromise patient dignity.

Modern mobile technology and artificial intelligence present opportunities to develop intelligent safety systems that can monitor patients unobtrusively while providing immediate alerts to caregivers. This paper introduces ZoneZap, a context-aware safety system that combines real-time location tracking and machine learning-based anomaly detection. The system supports two distinct roles: \textbf{users} (wards), who carry the app and may trigger panic alerts and receive reminders, and \textbf{guardians}, who link to one or more wards, receive panic and wandering alerts, and can create reminders for their wards.

The primary contributions of this work include:
\begin{itemize}
    \item A native Android (Kotlin) mobile application with dual user/guardian modes, real-time location tracking, emergency panic alerts, and a reminder system
    \item A cloud-based architecture using Firebase (Firestore, Cloud Functions, FCM) for real-time data processing and guardian notification
    \item Server-side anomaly detection in Cloud Functions (speed and movement variance) that triggers wandering alerts and notifies guardians
    \item An optional Python AI engine using the Isolation Forest algorithm for training and prediction on movement data
    \item Firestore security rules that enforce user and guardian access control over alerts, reminders, and movement logs
\end{itemize}

\section{Related Work}

Several existing systems address patient monitoring and safety. GPS tracking devices have been used for location monitoring, but they often lack intelligent analysis capabilities. Wearable devices with fall detection exist, but they may not detect wandering behavior early enough. Smart home systems provide indoor monitoring but are limited to fixed locations.

Recent research has explored machine learning for behavior analysis in healthcare. Isolation Forest algorithms have shown promise in detecting anomalies in time-series data, making them suitable for movement pattern analysis. However, few systems integrate real-time anomaly detection with a dedicated mobile application and a clear guardian--ward relationship for cognitive impairment safety.

\section{System Architecture}

\subsection{Overview}

ZoneZap consists of three main components: (1) a native Android mobile application for users and guardians, (2) a Firebase-based backend for data storage, processing, and push notifications, and (3) an optional Python AI engine for anomaly detection. The mobile app authenticates users via Firebase Authentication and stores profiles, alerts, reminders, and movement logs in Firestore. Cloud Functions react to new alerts and movement logs to perform anomaly checks and send FCM notifications to guardians or users.

\subsection{Mobile Application}

The native Android application is implemented in Kotlin using Material Design components. It supports two modes selected at login: \textbf{User} (ward) and \textbf{Guardian}.

\textbf{User (ward) flow:} After login, the user sees a Home screen showing current location (from the Fused Location Provider), upcoming reminders, and a panic button. Location updates are written to the Firestore \texttt{movement\_logs} collection with configurable interval and distance filters (e.g., 5-second interval, 10-meter minimum displacement). The Panic screen allows one-touch creation of an emergency alert document in the \texttt{alerts} collection; a Cloud Function then sends FCM notifications to all guardians linked in the user's profile. The Reminders screen allows listing, creating, editing, and marking reminders complete; a scheduled Cloud Function checks for overdue reminders every 5 minutes and sends FCM to the user.

\textbf{Guardian flow:} The Guardian screen allows adding wards by email (linking the guardian's UID to the ward's \texttt{guardians} array and the ward's UID to the guardian's \texttt{wards} array in Firestore). Guardians can create reminders for their wards and receive FCM notifications when a ward triggers a panic alert or when the system generates a wandering alert.

\textbf{Technical stack:} Android SDK, Kotlin, AndroidX, Material Components, Kotlin Coroutines, Firebase SDK (Auth, Firestore, FCM), and Fused Location Provider for GPS. Permissions include fine and coarse location, background location, and notifications.

\subsection{Cloud Backend}

The Firebase backend provides:

\textbf{Firestore Database:} Collections include \texttt{users} (with \texttt{type} ``user'' or ``guardian'', \texttt{guardians} and \texttt{wards} arrays, \texttt{fcmToken}), \texttt{alerts} (userId, alertType such as PANIC or WANDERING, location, timestamp, status), \texttt{reminders} (userId, title, description, scheduledTime, isCompleted, createdBy), and \texttt{movement\_logs} (userId, latitude, longitude, timestamp, speed, heading). Composite indexes support queries by userId and timestamp or status.

\textbf{Cloud Functions (Node.js 18):} Three functions are implemented:
\begin{itemize}
    \item \texttt{onEmergencyAlert}: Triggered on creation of a document in \texttt{alerts}. Fetches the user's \texttt{guardians} list, retrieves each guardian's \texttt{fcmToken}, and sends an FCM message with alert type and location. Also writes to \texttt{alert\_logs} for analytics.
    \item \texttt{analyzeLocationPatterns}: Triggered on creation of a document in \texttt{movement\_logs}. Fetches the last 30 movement logs for that user. If at least 10 exist, runs an in-function anomaly check: HIGH\_SPEED if average speed exceeds 5 m/s, and ERRATIC\_MOVEMENT if the variance of step distances exceeds a threshold. If anomalies are found, creates a new document in \texttt{alerts} with alertType WANDERING, which in turn triggers \texttt{onEmergencyAlert} and notifies guardians.
    \item \texttt{checkOverdueReminders}: Scheduled to run every 5 minutes. Queries \texttt{reminders} where \texttt{isCompleted} is false and \texttt{scheduledTime} is less than or equal to the current time, then sends FCM notifications to the corresponding users.
\end{itemize}

\textbf{Firebase Cloud Messaging (FCM):} Used to deliver push notifications to guardians (panic and wandering alerts) and to users (overdue reminders). Message payloads include alert type, location when applicable, and reminder title/description.

\subsection{AI Engine}

The optional Python-based AI engine uses the Isolation Forest algorithm (scikit-learn) to detect anomalous movement patterns. Features extracted from location data include distance from a reference (e.g., home) location, velocity, heading change, and temporal features (hour of day, day of week). The model can be trained on historical data exported from Firestore (\texttt{movement\_logs}) or from CSV files; training scripts (\texttt{train.py}, \texttt{train\_with\_firebase.py}) produce a serialized model (\texttt{model.pkl}) and a training report. The prediction script (\texttt{predict.py}) loads the model and can score new location points; anomalies are indicated by negative scores. This engine can be used for offline analysis or, in future work, integrated into the real-time pipeline (e.g., by invoking the model from a Cloud Function or a separate microservice).

\section{AI-Based Anomaly Detection}

\subsection{Isolation Forest Algorithm}

Isolation Forest is an unsupervised learning algorithm that identifies anomalies by isolating observations. It works on the principle that anomalies are few and different, making them easier to isolate than normal points.

For a dataset with $n$ samples, the algorithm:
\begin{enumerate}
    \item Randomly selects a feature and split value
    \item Partitions the data recursively
    \item Calculates the path length (number of splits) to isolate each point
    \item Anomalies require fewer splits to isolate
\end{enumerate}

The anomaly score is calculated as:
\begin{equation}
s(x,n) = 2^{-\frac{E(h(x))}{c(n)}}
\end{equation}
where $E(h(x))$ is the average path length, and $c(n)$ is the average path length for unsuccessful searches in a Binary Search Tree.

\subsection{Feature Engineering}

Location data is transformed into features suitable for anomaly detection:

\textbf{Distance from Home:} Euclidean distance from a predefined home location, converted to meters. Large distances may indicate wandering.

\textbf{Velocity:} Speed of movement in m/s. Unusually high speeds may indicate rapid movement or vehicle use.

\textbf{Heading Change:} Rate of change in direction. High variance indicates erratic movement patterns.

\textbf{Temporal Features:} Hour of day and day of week capture time-dependent patterns (e.g., wandering more likely at night).

\subsection{Model Training and Evaluation}

The Python model is trained on movement data (from Firestore or CSV), with configurable contamination (e.g., 5\%). Training produces \texttt{model.pkl} and a training report. Evaluation can be performed using standard metrics (accuracy, precision, recall, F1) on a held-out set. The in-cloud anomaly logic (Cloud Function) uses simpler rules (speed and variance thresholds) for real-time response; the Isolation Forest model can be used for batch analysis or future integration into the live pipeline.

\section{Implementation Details}

\subsection{Mobile Application}

The Android application is developed using:
\begin{itemize}
    \item Kotlin and Android SDK
    \item Material Design (Material Components)
    \item AndroidX (AppCompat, RecyclerView, etc.)
    \item Kotlin Coroutines for asynchronous operations
    \item Fused Location Provider for GPS tracking (high accuracy, configurable interval and distance)
    \item Firebase SDK: Authentication (email/password), Firestore, Cloud Messaging
\end{itemize}

Screens (Activities) include LoginActivity (with User/Guardian mode selection), HomeActivity (location, reminders, panic), PanicActivity, RemindersActivity, and GuardianActivity (wards, add by email, create reminders for wards). Services include UserService (profile and guardian--ward linking), LocationService (permissions and location flow), EmergencyService (create alerts), and ReminderService (CRUD for reminders). Data models map to Firestore documents (e.g., Reminder, EmergencyAlert, LocationData).

\subsection{Cloud Functions}

Firebase Cloud Functions are implemented in Node.js 18 in \texttt{backend/functions/index.js}:
\begin{itemize}
    \item \texttt{onEmergencyAlert}: Firestore trigger on \texttt{alerts/\{alertId\}} onCreate; sends FCM to guardians.
    \item \texttt{analyzeLocationPatterns}: Firestore trigger on \texttt{movement\_logs/\{logId\}} onCreate; fetches recent logs, runs \texttt{detectAnomalies} (Haversine distance, speed and variance thresholds), creates WANDERING alert if needed.
    \item \texttt{checkOverdueReminders}: Pub/Sub schedule every 5 minutes; queries overdue reminders and sends FCM to users.
\end{itemize}

\subsection{Security}

Firestore security rules ensure: (1) users can read/write their own profile and can be read/updated by linked guardians for linking; (2) alerts are created only with the requester's userId and read/updated by the owner or guardians of that user; (3) reminders are created by the owner or a guardian of the ward and read/updated accordingly; (4) movement logs are created only for the requester's userId and read by the owner or guardians; (5) \texttt{alert\_logs} is writable only by the server (no client access).

\section{Experimental Results}

\subsection{Dataset}

Experiments can be conducted using movement data from Firestore (\texttt{movement\_logs}) or simulated data representing normal daily routines, wandering episodes (distant or erratic movement), and rapid or unusual locations.

\subsection{Performance Metrics}

The Cloud Function-based anomaly logic provides immediate triggering when speed or variance thresholds are exceeded. Guardian notification latency depends on FCM and typically is on the order of seconds. The optional Isolation Forest model can be evaluated on held-out data for accuracy, precision, recall, and F1-score. System availability follows Firebase SLA when deployed on a Blaze plan.

\subsection{User Study}

A pilot study with users and guardians can assess: ease of use of the User and Guardian interfaces, time to add wards and create reminders, perceived usefulness of panic and wandering alerts, and reduction in response time to emergencies compared to manual monitoring.

\section{Discussion}

\subsection{Strengths}

ZoneZap provides:
\begin{itemize}
    \item Clear separation of user (ward) and guardian roles with secure linking
    \item Real-time panic and wandering alerts to guardians via FCM
    \item Automated reminder notifications via a scheduled Cloud Function
    \item Scalable Firebase backend and optional Python AI for anomaly detection
    \item Native Android implementation for performance and maintainability
\end{itemize}

\subsection{Limitations}

Current limitations include:
\begin{itemize}
    \item GPS accuracy and battery usage from location updates
    \item Indoor tracking limitations
    \item Requirement for smartphone ownership and basic literacy
    \item Privacy considerations for continuous location data
\end{itemize}

\subsection{Future Work}

Future enhancements may include: geofencing (safe/restricted zones with boundary alerts), a guardian web dashboard, wearable device integration, indoor positioning (e.g., Bluetooth beacons), deeper integration of the Python Isolation Forest model into the real-time pipeline, and multi-language support.

\section{Conclusion}

ZoneZap is a safety monitoring system for cognitively impaired users (wards) and their guardians. It combines a native Android app (Kotlin) with Firebase (Firestore, Cloud Functions, FCM) and an optional Python AI engine. Users benefit from location tracking, panic alerts, and reminders; guardians receive real-time panic and wandering alerts and can manage reminders for their wards. The architecture is scalable and secured by Firestore rules. Future work will focus on geofencing, richer AI integration, and expanded deployment.
